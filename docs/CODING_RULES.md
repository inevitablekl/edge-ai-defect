# CODING_RULES.md

## 1. Purpose

This document defines coding rules for:

**Edge AI Industrial Defect Detection Deployment and Optimization**

Chinese name:

**边缘 AI 工业缺陷检测部署与优化项目**

The rules apply to:

- Python training-side scripts.
- C++ deployment-side application.
- CMake build files.
- YAML configuration files.
- Experiment logging scripts.
- Code generated by Codex or other AI agents.

This project is an engineering deployment project, not an AI algorithm research project.

---

## 2. Core Coding Principles

Code must be:

- Simple.
- Modular.
- Readable.
- Testable.
- Reproducible.
- Easy to debug.
- Easy to explain in thesis defense and interviews.

Code must avoid:

- Overengineering.
- Hidden global state.
- Unnecessary abstraction.
- Large one-shot refactoring.
- Hardcoded experiment parameters.
- Silent error swallowing.
- Fake benchmark logic.
- Fake experiment output.
- GUI logic in core inference code.
- ROS2 runtime dependency in v1.
- INT8 logic unless explicitly approved.

---

## 3. Language Responsibility

Python is used for:

- Dataset conversion.
- Dataset split.
- YOLOv8n training.
- ONNX export.
- Experiment summary.
- Paper table / figure generation.

C++ is used for:

- YAML configuration loading.
- Image / video input.
- Preprocessing.
- ONNX Runtime inference.
- TensorRT FP16 inference.
- YOLOv8 postprocessing.
- Profiling.
- Result output.
- SerialRunner.
- PipelineRunner.
- Command-line deployment.

Python must not become the main deployment runtime.

C++ must not contain training logic.

---

## 4. C++ Standard and Build

The C++ project should use:

```text
C++17
```

The build system must use:

```text
CMake
```

Expected build flow:

```bash
mkdir -p build
cd build
cmake ..
make -j$(nproc)
```

If C++ standard or build method changes, record the reason in:

```text
docs/personal/DECISIONS.md
```

---

## 5. Repository Layout Rules

C++ headers should be placed under:

```text
include/edge_ai_defect/
```

C++ source files should be placed under:

```text
src/
```

Python scripts should be placed under:

```text
scripts/
```

YAML configuration files should be placed under:

```text
configs/
```

Experiment logs and outputs should be placed under:

```text
experiments/
```

Do not place source code directly in random directories.

Do not mix generated artifacts with source code.

---

## 6. C++ Module Boundaries

The deployment-side C++ code should be organized around:

| Module            | Responsibility                               |
| ----------------- | -------------------------------------------- |
| ConfigManager     | YAML loading and validation                  |
| FrameSource       | Image sequence, video, optional camera input |
| Preprocessor      | Image to model input buffer                  |
| InferenceEngine   | Abstract inference backend interface         |
| ONNXRuntimeEngine | ONNX Runtime baseline                        |
| TensorRTEngine    | TensorRT FP16 backend                        |
| PostProcessor     | YOLOv8 decode, threshold, NMS                |
| Profiler          | Latency, FPS, resource logs                  |
| ResultSink        | Detection result and visualization output    |
| SerialRunner      | Sequential baseline runtime                  |
| PipelineRunner    | Multi-thread runtime                         |
| common            | Shared structs and utilities                 |

Backend-specific logic must stay inside backend modules.

Runner logic must not contain ONNX Runtime or TensorRT-specific implementation details.

---

## 7. Naming Rules

Use clear and descriptive names.

Recommended C++ style:

* Classes: `PascalCase`
* Functions: `snake_case`
* Variables: `snake_case`
* Constants: `kPascalCase` or `UPPER_SNAKE_CASE`
* Files: `snake_case.hpp`, `snake_case.cpp`

Examples:

```cpp
class ConfigManager;
class ONNXRuntimeEngine;
class TensorRTEngine;

bool load_config(const std::string& path);
double inference_latency_ms;
```

Avoid vague names:

```cpp
data
tmp
foo
process_all
do_something
```

---

## 8. Configuration Rules

Experiment parameters must come from YAML configuration.

Do not hardcode:

* backend
* runtime mode
* model path
* TensorRT engine path
* input source
* input size
* confidence threshold
* NMS threshold
* output directory
* log directory
* warmup frame count
* measured frame count
* pipeline queue size
* pipeline drop policy

Invalid configuration must fail with a clear error message.

---

## 9. Error Handling Rules

Errors must be explicit and actionable.

Good examples:

```text
Config file not found: configs/infer_trt_serial.yaml
Missing required config field: runtime.backend
Model file not found: models/onnx/yolov8n_640.onnx
TensorRT engine file not found: models/trt/yolov8n_640_fp16.engine
Unsupported backend: rknn
Unsupported runtime mode: async
Invalid input size: [500, 500]
```

Bad example:

```text
Something went wrong.
```

Do not silently ignore critical errors.

Do not continue after failed model loading, failed config loading, or failed backend initialization.

---

## 10. Dependency Rules

Allowed or expected C++ dependencies:

* OpenCV
* yaml-cpp
* ONNX Runtime C++ API
* TensorRT
* CUDA Runtime

Allowed or expected Python dependencies:

* ultralytics
* torch
* opencv-python
* numpy
* pyyaml
* pandas
* matplotlib

Optional test dependency:

* GoogleTest

Do not introduce heavy dependencies without explicit approval.

If a new dependency is required, explain:

* why it is needed
* which module uses it
* whether it is required or optional
* whether it affects Jetson deployment
* whether a lighter alternative exists

---

## 11. Logging Rules

The system must produce logs suitable for experiment reproduction.

Logs should include:

* timestamp
* git commit if available
* device
* backend
* runtime mode
* model name
* input size
* precision mode
* source type
* warmup frames
* measured frames
* latency statistics
* FPS
* resource metrics if available
* notes

Output formats:

```text
CSV
JSON
```

If a metric is unavailable, record:

```text
Not available
```

Do not fabricate missing metrics.

---

## 12. Profiling Rules

Profiler must support:

* preprocessing latency
* inference latency
* postprocessing latency
* total latency
* FPS

Profiler should support when available:

* CPU usage
* GPU usage
* memory usage

Warmup frames should be excluded from final statistics.

Pipeline mode must distinguish:

* throughput / FPS
* single-frame latency
* end-to-end latency

Do not claim pipeline mode always reduces latency.

---

## 13. Testing Rules

The project should include basic tests for:

* YAML config loading
* invalid config detection
* preprocessing output shape
* postprocessing basic behavior
* profiler CSV / JSON output

Every major feature should provide:

* build command
* run command
* input requirement
* expected output
* failure behavior

Codex-generated code should not be accepted without verification.

---

## 14. Python Script Rules

Python scripts should be placed under:

```text
scripts/
```

Python scripts should:

* use clear command-line arguments
* avoid hardcoded absolute paths
* write records for important outputs
* fail clearly on missing files
* avoid changing dataset labels silently
* avoid generating fake experiment data

Dataset and training scripts must keep outputs reproducible where possible.

---

## 15. C++ Runtime Rules

The C++ application must support command-line execution.

Expected usage:

```bash
./edge_ai_defect --config configs/infer_trt_serial.yaml
```

The application should:

* parse config path
* load YAML
* validate config
* create modules
* select backend
* select runtime mode
* run inference
* save results
* save logs

The application must not require source code modification to switch experiments.

---

## 16. AI Agent Rules

Codex and other AI agents must work in small tasks.

Good task:

```text
Implement ConfigManager to load YAML and validate required fields.
```

Bad task:

```text
Build the whole project.
```

AI agents must not:

* rewrite unrelated files
* perform large refactors without approval
* introduce heavy dependencies without explanation
* invent experiment results
* add GUI / ROS2 / INT8 / RKNN by themselves
* silently change architecture
* mix training logic into deployment runtime

The project owner must review diffs and understand key modules.

---

## 17. Experiment Integrity Rules

AI agents and scripts must never fabricate:

* Precision
* Recall
* mAP
* FPS
* latency
* CPU usage
* GPU usage
* memory usage
* power usage
* stability results
* paper tables
* paper figures

If data is unavailable, write:

```text
TBD: real experiment data required
```

or:

```text
Not measured yet
```

Paper conclusions must be based on real measured logs.

---

## 18. Final Rule

Prefer:

* simple code over clever code
* clear interfaces over excessive abstraction
* real logs over fake tables
* reproducible experiments over one-time demos
* working system over large unfinished design
